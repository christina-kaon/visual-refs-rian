EMOCHI · MODEL DEV PRIORITY + TRAINING ROADMAP · v0.5 · 2026-05-25

生图模型理想态 · 开发优先级 + 训练思路

基于 Rian Q/C/A wiki + emochi top100 数据 + keyword verify 综合判断 · v0.5 · base 是图像编辑模型 (非 T2I)（强化学习 + 深度生成模型）
核心原则：所有理想态不可能一次完成。按 Quality (基础门槛) → Content (核心覆盖) → Aesthetics P0 (商业大头) → Aesthetics P1 (扩量储备) → P2 (长尾兜底) 分 5 阶段。

核心数据 anchor：top100 真人 OC Real 52 + Multi 26 + Celebrity 6；top100 动漫 OC Virtual 36 + Multi 26 + marriage 15 + OC Real 15。keyword verify：真人 prompt 99.7% 含 cinematic（baseline 而非流派）；3D / GQ / studio / game_art 几乎 0（确认 P3 砍）。

5 阶段开发优先级

Phase 0 · P0

Quality 基础门槛（任何模型都得过）

WHY：不过这 5 项的图直接是废片，用户第一秒就出戏 —— 没有任何 Aesthetic / Content 能补救。Quality 是所有后续训练的 gate condition

项级别训练思路
D1 整体不糊P0pre-train + SFT 阶段用高分辨率 + 锐度增强数据，反例 reject sampling
D2 解剖学不崩 / 五官比例对P0手部专项数据增强（手部 ControlNet / DWPose 标注大量人手）+ 反例 dataset 训练 anatomy classifier reward
D3 空间逻辑不错乱P0多人 + 多物体场景的物理一致性专项数据，特别强化 "持握" "坐立" "推拉" 这类 interaction
D4 不出油 / 不塑料P1皮肤质感专项 LoRA + 磨皮反例剔除（重点剔训练集里的修图过度样本）
D5 人物 vs 背景不撕裂P1character-scene 统一打光数据 + 端到端训练（不能是 character 单独训完再贴背景）

资源：一次性基础投入。pre-train + 通用 SFT，所有后续 sub-spec 受益。Quality 不分真人 / 动漫，是统一 gate

Phase 1 · P0

Content 核心能力（emochi top100 直接 verify 的需求）

WHY：真人 + 动漫两端 chats 大头的 trope / scene 必须覆盖，否则模型只能"画一个好看的脸"但生成不了"剧情高光时刻"

项级别训练思路
Trope 矩阵P0mafia / werewolf / CEO / 暗黑男主 / 萌系女角色 / K-pop celebrity / 校园暴力恋情。每个 trope 配 200-500 张 high-quality 样本
Scene 矩阵P0雨夜街道 / 车内 / 顶层公寓 / 教堂 / 卧室 / 街道 / 异世界 / dungeon / fantasy world / 学院 — 场景与情绪的咬合训练
Shot angle 多样化P0低机位仰视 / 高机位俯瞰 / 侧 3/4 / 镜子自拍 / 背影回眸 — 当前模型 default "正面平视中景" 是最大问题
双人 / 群像构图P0壁咚 / 公主抱 / 拥抱 / 对峙 — 力的传递 + 关系温度。这块 emochi 现状模型最弱
NSFW 嫁接 cinematic 基底P1真人 18.7% NSFW chats 占比。NSFW 训练必须在 cinematic dim lighting 基底上，不在 candid 软光基底上
情绪光谱P15 类情绪（脆弱破碎 / 危险诱惑 / 纯真无害 / 狂气中二 / 日常温柔）+ "表情和场景必须咬合"约束

资源：中等投入。trope × scene × shot_angle 三维 stratified 5000-10000 张精选 SFT 样本。可以 cross-medium 共享 caption 标准

项级别训练思路

真人 B · Dark Romance 危险风P0黑帮电影截图 + GQ Korea + 朴赞郁韩剧黑帮 + Helmut Newton。cinematic 已 baseline (99.7% prompt 含)，重点训电影质感 vs 业余仿冒

资源：高投入。每 sub-spec 需 200-1000 张美术总监级 curation 的 reference + 反锚点剔除工作量。这是 emochi 真正的差异化

Phase 3 · P1

Aesthetics 风格轨道 (P1 流派 - 扩量储备)

WHY：国内 / 东亚扩量必备，但现有海外用户需求弱。提前训好作为产品扩量的视觉储备

项级别训练思路

Phase 4 · P2

Long tail 兜底（P2，小众粘性）

WHY：不是主流，但能体现模型审美覆盖面。当 P0+P1 跑稳后，作为 "model 能力测试集" 的 edge case，不投资源训练

项级别训练思路

训练 Pipeline（4 阶段 post-training）

基于 emochi 现状能跑出的 4 阶段后训练 pipeline：

Stage 1: Base Pre-train + General SFT（Quality 阶段）

  • Base 选型（v0.5 重大修正）：emochi 用的是开源图像编辑模型 (image editing model)，不是 pure T2I。这一点对训练 pipeline 影响深远 — 见原则 9。
  • 候选开源 image editing model：FLUX.1 Kontext（黑森林 2025，当前 SOTA）/ OmniGen / OmniGen2（北大智源） / Step1X-Edit（字节） / BAGEL / InstructPix2Pix 系（早期 baseline）。需 Rian 确认现在 emochi 用的是哪个
  • Image editing model 接受 (reference image + prompt) 输入，输出 (ref-style-preserved 新场景图)。比 T2I 多一个 ref-image conditioning channel
  • 不要自己 from scratch（北航 deck 也明确：站在开源 base 之上）
  • General SFT 数据：emochi 现有真实生图 + caption 重写（重写很重要，让 prompt 描述 align 内容生态）
  • Quality gate：用 1165 张反例库做 reject sampling — 训练集任何样本如果命中 D1-D5 任一维度，直接剔除
  • 验收标准：5 个 Quality 维度 VLM judge ≥ 3/4（hard penalty 阈值）

Stage 2: Content Coverage SFT

  • 数据准备：trope × scene × shot_angle 三维 stratified sample 5000-10000 张高质量样本，每格至少 5 条
  • caption 模板化：每个 trope 配标准化 caption 模板（emochi 后端 prompt 模板已经做了一半）
  • 训练方式：full SFT，因为是 base model 能力扩展，不是风格变化
  • 验收标准：能从 prompt 生成对应 trope/scene/angle 的图，不是 default "正面中景"

Stage 3: Aesthetics Sub-spec Full SFT（v0.3：纯 SFT，不用 LoRA）

  • 每个 P0 sub-spec 独立 full SFT 模型：6 个 P0 (真人 2 + 动漫 4) = 6 个独立 model checkpoint
  • WHY not LoRA：emochi sub-spec 之间美学是相反方向（A 轨软光 vs B 轨硬光对立；乙女精致 vs 抖红美颜冲突）。LoRA 在 base 上叠加这种"反向"风格时容易回归到 base 的中庸输出，无法真正"长成"该流派分布。Full SFT 才能让模型完整重塑输出分布
  • v0.5 修正：face encoder / IP-Adapter 模块冻结。SFT 只动 style decoder + cross-attn 部分。否则 sub-spec 训练会破坏 character likeness — 用户 ref image 喂进去出来的角色脸都变了
  • 每个 SFT 训练数据：5K-20K 张美术总监级 curation 的样本 + ~2K-5K 反锚点 negative samples（注意：纯 SFT 比 LoRA 需要的数据量大 10-20×）
  • 关键约束：dataset 必须严格按谱系筛选 + 反锚点剔除强度比 LoRA 时代要高 — 因为 SFT 学得更深，污染更难恢复
  • 资源 cost：6× full SFT (每个 ~base model 的 1-3% step 数)。比 LoRA 方案高 5-10×，但风格强度上限更高、可单独 deploy / A-B test
  • 验收标准：sub-spec 用户群盲选（不能用 VLM 评，要找真实的 K-pop fan / 乙女游戏粉丝盲测）

Stage 4: RLHF / DPO 偏好对齐（v0.4 update — 吸收北航 RL deck）

  • 方法选型：Diffusion-DPO（不是 LLM-PPO / GRPO 直接用），因为 image gen 的 reward 是 trajectory-level（整张图打分）而不是 token-level。Diffusion-DPO 是 image 端对应物，结构 align 北航 deck 里 LLM 的 DPO/GSPO 思路 — 但落到 latent space
  • 借鉴 GRPO 思路（节约 value model 内存）：对同一个 prompt 采样 N 个图，按用户偏好做 group-relative 排序而不是绝对 reward。省一个 value model 的内存/算力（北航 deck 6.2.4.3）
  • 借鉴 DAPO 思路（防 entropy collapse）：sub-spec 用户偏好窄，DPO 训练时容易让"原本低概率风格"被过早剪掉。**asymmetric clip (ε_low=0.2, ε_high=0.28)** 防止 entropy collapse — emochi sub-spec 这种"风格深度训练"特别需要（北航 deck 6.2.4.4）
  • 数据来源：emochi 用户实际偏好信号（chat 深度 / 留存 / 付费率 / 用户对生成图主动 thumbs-up）—— 构造 preference pairs
  • preference pairs 构造：在同一 prompt + dialogue context 下，对每个 prompt 采样 N=4-8 张候选图，按 (留存 + chat 深度) 排序，取 win / loss pair 做 DPO 训练
  • cost 最高但收益最 emochi-specific：这一步把通用美学拉到 emochi 用户偏好分布
  • 验收标准：上线 A/B test，看新模型 vs 旧模型的留存 / chat 深度提升

Stage 5: Evaluation Loop (持续)

  • 100 条 calibration set（李承燕剩 70 条标完）→ 调 hard penalty 阈值
  • 每月 freshness check：跑一遍 calibration，看模型有没有 regression
  • 新 trope/scene 增补：用户 chat 内容里的新热点（如新 IP 上线）触发 incremental SFT

关键原则（v0.2 升级）

原则 6：训练 data 是 distribution 的 sample，不是 product spec 的镜像。"我们 ship 哪些流派" 和 "我们用什么 data 训练" 是两条独立的决策线 — 前者是产品定义，后者是数据工程。混淆这两个会导致两种错误：(a) 数据过窄导致模型欠拟合（只看过 emochi 风格图就训不出 baseline 质感）；(b) 数据过泛导致模型走偏（什么都喂就会 mode collapse 到"中庸"）

关键原则（v0.5 新增 — base 是 image editing model 的 implication）

原则 9（新）：Image editing model 是 emochi 的真正 base，character consistency 不靠模型记忆而靠 ref image IP injection。这跟纯 T2I 的训练策略有 4 个本质区别：

  • 训练样本是三元组而不是 pair：(reference image, prompt, generated image)。每条训练样本必须有匹配的 ref image。emochi 100 测试集 CSV 里 ref_image_path 字段就是这个角色 — duidui 之前要 ref + generated_image 这两列是对的
  • character likeness 不靠模型 finetune 记住具体 face：靠 ref image IP injection 机制（IP-Adapter / image cross-attn / FaceID encoder）。这意味着 sub-spec SFT 训练里 face IP 模块要冻结，只训风格部分。冻错地方会让 face likeness 崩
  • D1 角色一致性不是看模型有没有记住角色，是看 ref-image IP injection module 是否被破坏。Quality D1 评估 prompt 要 frame 为"生成图和 ref image 的 face match"，不能 frame 为"角色一致性"
  • emochi prompt 模板里 "Follow the reference image style exactly. The character's face must match" 是直接证据 — 这就是 image editing model 的 SOP 用法

原则 10（新）：Aesthetics sub-spec model 仍然是 6 个独立 image editing model，但每个的 ref-IP 模块共享（不要重训 face encoder）。

  • 6 个 P0 sub-spec → 6 个 base image editing model 的 SFT 分叉，但 face encoder / IP-Adapter 部分冻结复用
  • 训练 cost 不会 6× — 而是 6× (style decoder + cross-attn) ≈ 2-3× full model retraining cost
  • 这是个好消息：原则 5 里 "纯 SFT cost 上升 5-10×" 在 image editing model setting 下，实际更接近 2-3×

关键原则（v0.4 新增 — 来自北航深度生成模型 deck）

原则 7（新）：Diffusion model is secretly a zero-shot classifier。训出的 sub-spec model 本身就是个 "这张图是不是 X 流派" 的 classifier — 利用 ELBO + Bayes 公式可以直接 zero-shot 分类（北航 deck "关键论文：生成构筑理解"段）。这给我们三个 emochi-specific 红利：

  • 训练完即得 classifier：不用额外训 style classifier，sub-spec model 本身就是。可以反过来给 benchmark VLM judge 提供 ground-truth anchor — 解决了我之前 brief 里"D3 审美 Pearson 0.34 不可靠"的 root cause
  • 反锚点剔除可以自动化：用 stage 3 训完的 model 自动 scan 训练集，找 ELBO loss 偏高的样本 = 不符合 sub-spec 分布 = 反锚点候选。比纯人工剔除快 10-100×
  • style transfer cost 大幅降：sub-spec model A → B 的迁移可以用 model A 的 ELBO 作为 loss 引导，比从 base 重训省一半算力

原则 8（新）：Quality 层的"不出戏"哲学和 GAN 的 D-loss 是同构的。北航 deck 讲 GAN 时强调"逐点 L2 重构损失没考虑全局性，导致边缘细节差"。这跟你 wiki 文档里写 Quality 是"用户第一秒会不会出戏"完全 align —— 都是感知层面的全局一致性而不是 pixel-level error。Quality VLM judge 应该按"是否出戏"这个 perceptual 标准 prompt，而不是 pixel-level metric

5 个关键训练原则

1. Quality 不分流派 / Aesthetics 必须分流派

Quality 是底层物理 (anatomy / lighting / sharpness)，全 sub-spec 共享 base。Aesthetics 是用户审美偏好，分裂程度极高，必须独立 model 不能 LoRA-switch。

2. Cross-medium grammar 共享但 user preference 不共享

真人 A ↔ 动漫副（软光氧气）/ 真人 B ↔ 动漫主（硬光危险）共享 lighting grammar，可以共享训练 rubric。但用户偏好不重合 — 不能假设"喜欢真人 A 就也喜欢动漫副"。

3. 美术总监 curate dataset = 不可外包的核心工作

这是 emochi 跟通用 T2I 的真正差异化。crowdsource / web scraping 一把梭训不出"懂 fandom 语境"的模型 — Phase 2 / Phase 3 必须配美术总监 in-the-loop。

4. 反锚点剔除比正样本筛选更难

"哪些样本不该进训练集"通常比"哪些该进"更费判断力。我之前 brief 里"反锚点"那段就是这个目的 — 这部分 dataset curation 比例应该是 30% 时间正样本 + 70% 时间反锚点。

5. cinematic 是 default 不是流派的反向教训

当 99.7% 真人 prompt 都有同一个关键词，那个关键词不再是差异化信号。建议产品端定期 audit emochi 后端 prompt 模板，避免把"baseline 能力"和"流派"混淆。
